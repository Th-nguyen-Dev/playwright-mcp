# MUX-001: Remove `--headless` CLI flag that defeats Xvfb anti-detection strategy

## Metadata
| Field | Value |

## Description

When headless mode is requested, `buildArgs()` pushes the `--headless` CLI flag to the child `@playwright/mcp` process AND creates a config file with `headless: false`. The intent is that Chrome should run headed on an Xvfb virtual display (avoiding bot detection). But the `@playwright/mcp` CLI resolves config with precedence: `default -> configFile -> env -> cliFlags`. CLI flags override config, so `--headless` wins and Chrome runs in actual headless mode. The Xvfb display sits unused, wasting resources and defeating the entire anti-detection strategy.

This is a **critical** bug because:
1. Every "headless" instance runs Chrome's headless mode, which is trivially detectable by anti-bot systems
2. Xvfb displays are allocated but never used (resource leak)
3. The config file comment explicitly documents the anti-detection intent, but the CLI flag contradicts it

## Technical Approach

- **Architecture Layer**: Backend
- **Design Patterns**: Configuration precedence, defense-in-depth
- **Implementation Strategy**:
  1. Remove `args.push('--headless')` on line 244 of `instance-manager.ts`. The config file's `headless: false` is the correct mechanism -- Chrome will run headed but on an invisible Xvfb display.
  2. Remove the unused `headless` parameter from `createLaunchConfig()` (line 281). The method already unconditionally writes `headless: false`, so the parameter is misleading. This also resolves MUX-009.
  3. Update the `createLaunchConfig` call site (line 269) to remove the `headless` argument.
  4. Verify that the Xvfb allocation path in `create()` still correctly gates on the `headless` flag (lines 120-123) -- it should, since the local variable `headless` is still computed but just not passed to `buildArgs` as a CLI flag.

## Implementation Notes

The key insight is lines 242-244 vs 299-305:

```typescript
// line 242-244 (REMOVE the --headless push)
const headless = instanceConfig.headless ?? this.config.defaultHeadless;
if (headless)
  args.push('--headless');  // <-- DELETE THIS LINE

// line 299-305 (config file -- this is CORRECT, keep as-is)
const config = {
  browser: {
    launchOptions: {
      headless: false,  // Always headed -- Xvfb makes it invisible
      args: launchArgs,
    },
  },
};
```

After the fix, the `headless` variable in `buildArgs()` is still used for Xvfb allocation (line 120-123) but is no longer incorrectly forwarded as a CLI flag.
