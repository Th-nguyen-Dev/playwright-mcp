## Description

`handleProxyTool()` in `tool-router.ts` has inconsistent error handling. There are three error paths:

1. **Missing `instanceId`** (line 48-53): Returns `{ content: [...], isError: true }` -- correct
2. **`getOrThrow(instanceId)`** (line 55): Throws a raw `Error` -- becomes a JSON-RPC protocol error (-32603 InternalError)
3. **`instance.client.callTool()`** (line 60-63): Can throw if child process has crashed -- also becomes a protocol error

MCP clients handle these two error types completely differently:
- **Protocol errors** (-32603) cause exceptions in the client SDK, often breaking the entire tool call flow
- **Tool errors** (`isError: true`) are handled gracefully and shown to the user as normal tool output

The inconsistency means "instance not found" crashes the client SDK while "missing instanceId" is handled gracefully, despite both being user-recoverable errors.

## Technical Approach

- **Architecture Layer**: Backend
- **Design Patterns**: Error normalization, defensive programming
- **Implementation Strategy**:
  1. Wrap the entire body of `handleProxyTool()` in a try/catch block
  2. In the catch block, return a proper `ToolCallResponse` with `isError: true`
  3. This ensures all errors from `getOrThrow()` and `callTool()` are returned as tool responses, not protocol errors

## Implementation Notes

Replace the current `handleProxyTool()` with:

```typescript
private async handleProxyTool(name: string, args: Record<string, unknown>): Promise<ToolCallResponse> {
  const instanceId = args.instanceId as string | undefined;
  if (!instanceId) {
    return {
      content: [{ type: 'text', text: 'Missing required parameter: instanceId' }],
      isError: true,
    };
  }

  try {
    const instance = this.instanceManager.getOrThrow(instanceId);

    // Strip instanceId before forwarding to child
    const { instanceId: _, ...childArgs } = args;

    const result = await instance.client.callTool({
      name,
      arguments: childArgs,
    });

    return result as ToolCallResponse;
  } catch (error) {
    return {
      content: [{
        type: 'text',
        text: `Error calling ${name} on instance "${instanceId}": ${error instanceof Error ? error.message : String(error)}`,
      }],
      isError: true,
    };
  }
}
```

Note: The `handleAuthExport()` method on line 168 has the same pattern (calls `getOrThrow` inside try/catch) and already handles it correctly. This fix brings `handleProxyTool` in line with that pattern.
