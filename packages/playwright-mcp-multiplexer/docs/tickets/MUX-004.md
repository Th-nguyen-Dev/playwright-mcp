# MUX-004: Add discovery mutex to prevent concurrent `discoverTools()` races

## Description

Both `ListToolsRequestSchema` and `CallToolRequestSchema` handlers in `multiplexer-server.ts` check `!this.toolRegistry.isInitialized()` and call `this.discoverTools()`. If two MCP requests arrive before discovery completes, two probe instances get spawned and `ToolRegistry.discoverTools()` is called concurrently without synchronization. The `proxyTools`, `allTools`, and `initialized` fields are set without atomicity.

**Consequences**:
- Two probe instances spawned instead of one (resource waste, extra Xvfb display allocation)
- Race condition on `ToolRegistry` fields: `proxyTools` could be set by one call, then overwritten by the other mid-read
- The second probe instance may not be cleaned up if the first `discoverTools()` sets `initialized = true` before the second completes

## Technical Approach

- **Architecture Layer**: Backend
- **Design Patterns**: Mutex/once pattern via cached Promise
- **Implementation Strategy**:
  1. Add a `private discoveryPromise: Promise<void> | null = null` field to `MultiplexerServer`
  2. Replace the inline `!this.toolRegistry.isInitialized()` checks with a call to `ensureToolsDiscovered()`
  3. `ensureToolsDiscovered()` caches the discovery promise so concurrent callers await the same operation

## Implementation Notes

Add to `MultiplexerServer`:

```typescript
private discoveryPromise: Promise<void> | null = null;

private async ensureToolsDiscovered(): Promise<void> {
  if (this.toolRegistry.isInitialized()) return;
  if (!this.discoveryPromise) {
    this.discoveryPromise = this.discoverTools().catch(err => {
      // Allow retry on failure
      this.discoveryPromise = null;
      throw err;
    });
  }
  return this.discoveryPromise;
}
```

Then update the handlers:

```typescript
// ListToolsRequestSchema handler
if (!this.toolRegistry.isInitialized()) {
  await this.ensureToolsDiscovered();
}

// CallToolRequestSchema handler -- same pattern
```

The error-reset pattern (`catch -> null`) is important: if the probe instance fails to spawn (e.g., Xvfb not installed), the next request should retry rather than permanently failing.
