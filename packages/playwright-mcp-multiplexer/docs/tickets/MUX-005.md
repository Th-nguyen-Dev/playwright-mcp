## Description

`findFreeNum()` in `virtual-display.ts` only checks the internal `displays` map to find a free display number. It does not check whether the display number is already in use system-wide (e.g., from another multiplexer instance, a CI runner, or an existing X server).

**Consequences**:
- If display `:10` is already in use by another process, `Xvfb` spawn fails with a confusing error message
- Worse case: `waitForLock()` could succeed immediately on the existing lock file (written by the other process), causing Chrome to connect to the wrong display owned by another user/process
- In CI environments with multiple parallel test runners, display collisions are highly likely

## Technical Approach

- **Architecture Layer**: Backend / Infrastructure
- **Design Patterns**: Resource allocation with system-wide conflict detection
- **Implementation Strategy**:

  **Option A (Recommended): Check lock file existence before allocation**
  1. In `findFreeNum()`, check for `/tmp/.X<N>-lock` existence before returning a display number
  2. If the lock file exists, skip that number and try the next one
  3. This is simple and handles most collision cases

  **Option B: Use Xvfb `-displayfd` for kernel-assigned display**
  1. Use `Xvfb -displayfd <fd>` to let the kernel assign a free display number
  2. Read the assigned number from the file descriptor
  3. This is more robust but requires more complex fd management

  Option A is recommended for simplicity and sufficient correctness.

## Implementation Notes

```typescript
private findFreeNum(): number {
  for (let n = DISPLAY_RANGE_START; n <= DISPLAY_RANGE_END; n++) {
    // Skip numbers used by this manager
    if (this.displays.has(n)) continue;
    // Skip numbers used system-wide (lock file = X server is running on that display)
    if (fs.existsSync(`/tmp/.X${n}-lock`)) continue;
    return n;
  }
  throw new Error('No free virtual display slots available (:10-:99 all in use)');
}
```

Note: There is still a TOCTOU race between checking the lock file and spawning Xvfb, but this is acceptable because Xvfb itself will fail to start if the display is taken, and the error will propagate via `spawnXvfb()`. The check is an optimization to avoid unnecessary spawn attempts.
