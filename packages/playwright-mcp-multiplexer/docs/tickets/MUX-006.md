## Description

In `spawnXvfb()` (lines 58-77 of `virtual-display.ts`), there is a race condition between the `exit` event handler and the `waitForLock().then()` callback.

**Race scenario**:
1. Xvfb spawns and creates its lock file at `/tmp/.X<N>-lock`
2. Xvfb immediately exits (e.g., GPU driver error, signal from another process)
3. `waitForLock()` resolves because the lock file exists
4. The `exit` handler fires and calls `reject()`
5. The `.then()` callback runs, calls `this.displays.set(num, proc); resolve()`
6. Both `reject()` and `resolve()` fire on the same Promise

**Consequences**:
- The Promise constructor only honors the first settlement, but the `displays` map still gets the dead process registered
- Chrome is told to use `DISPLAY=:<N>` which points at a dead Xvfb -- rendering fails silently or crashes
- The dead process entry in the map prevents cleanup and blocks the display number from being reused

## Technical Approach

- **Architecture Layer**: Backend / Infrastructure
- **Design Patterns**: Promise settlement guard, liveness verification
- **Implementation Strategy**:
  1. Add a `settled` boolean flag to prevent double resolution of the Promise
  2. After `waitForLock()` resolves, check that the Xvfb process is still alive before registering it
  3. If the process has already exited, reject with a clear error

## Implementation Notes

```typescript
private spawnXvfb(num: number): Promise<void> {
  return new Promise((resolve, reject) => {
    let settled = false;
    const settle = (fn: () => void) => {
      if (!settled) { settled = true; fn(); }
    };

    const proc = spawn('Xvfb', [`:${num}`, '-screen', '0', '1920x1080x24', '-ac'], {
      stdio: 'ignore',
      detached: false,
    });

    proc.on('error', (err) => settle(() => reject(err)));

    proc.on('exit', (code) => {
      settle(() => reject(new Error(`Xvfb :${num} exited before ready (code ${code})`)));
    });

    this.waitForLock(num, 5000)
      .then(() => {
        // Re-check liveness: Xvfb may have exited between lock creation and now
        if (proc.exitCode !== null) {
          settle(() => reject(new Error(`Xvfb :${num} exited after creating lock (code ${proc.exitCode})`)));
        } else {
          settle(() => { this.displays.set(num, proc); resolve(); });
        }
      })
      .catch(err => {
        proc.kill();
        settle(() => reject(err));
      });
  });
}
```

Key changes:
1. `settled` flag prevents double settlement
2. `proc.exitCode !== null` check after lock file appears verifies liveness
3. All paths go through `settle()` to guarantee single settlement
