# MUX-010: Prevent double Ctrl+C from interrupting graceful shutdown

## Description

The signal handlers in `cli.ts` (lines 44-51) do not guard against re-entrant invocation:

```typescript
process.on('SIGINT', async () => {
  await server.close();
  process.exit(0);
});
process.on('SIGTERM', async () => {
  await server.close();
  process.exit(0);
});
```

If a second SIGINT arrives while the first handler is awaiting `server.close()`, Node.js invokes the handler again. The second handler calls `process.exit(0)` immediately (since `server.close()` may throw or hang on the second invocation), killing the process before cleanup completes.

**Consequences**:
- Orphaned Chrome and Xvfb processes
- Temporary profile directories not cleaned up
- Config files left in `/tmp/pw-mux/`

## Technical Approach

- **Architecture Layer**: Infrastructure / CLI
- **Design Patterns**: Signal handler idempotency, shutdown flag
- **Implementation Strategy**:
  1. Add a `shuttingDown` boolean flag, initially `false`
  2. On first signal, set `shuttingDown = true` and proceed with cleanup
  3. On subsequent signals while `shuttingDown` is true, ignore them (or optionally log "shutdown in progress")

## Implementation Notes

```typescript
let shuttingDown = false;

async function shutdown() {
  if (shuttingDown) return;
  shuttingDown = true;
  await server.close();
  process.exit(0);
}

process.on('SIGINT', shutdown);
process.on('SIGTERM', shutdown);
```

Alternative approach: remove the signal handlers entirely and let the MCP transport's close event trigger cleanup. But explicit signal handling is safer for ensuring Xvfb/Chrome cleanup.
