## Description

`isProxyTool(name)` in `tool-registry.ts` (lines 94-96) uses a linear scan:

```typescript
isProxyTool(name: string): boolean {
  return this.proxyTools.some(t => t.name === name);
}
```

This is O(n) per call, where n is the number of proxy tools (~30+ from `@playwright/mcp`). The method is called on every `CallToolRequestSchema` request (line 17 of `tool-router.ts`), making it a hot path.

While the performance impact is negligible for a typical MCP session (requests are human-paced), this is a code quality issue: there is already a `Set<string>` pattern used for management tools (`MANAGEMENT_TOOL_NAMES` on line 62), so proxy tools should follow the same pattern for consistency.

## Technical Approach

- **Architecture Layer**: Backend
- **Design Patterns**: Hash set lookup, consistent naming patterns
- **Implementation Strategy**:
  1. Add a `private proxyToolNames = new Set<string>()` field to `ToolRegistry`
  2. Populate it during `discoverTools()` alongside `proxyTools`
  3. Change `isProxyTool()` to use `this.proxyToolNames.has(name)` (O(1))

## Implementation Notes

```typescript
export class ToolRegistry {
  private proxyTools: Tool[] = [];
  private proxyToolNames = new Set<string>();  // NEW
  private allTools: Tool[] = [];
  private initialized = false;

  // ...

  async discoverTools(probeClient: Client): Promise<void> {
    const result = await probeClient.listTools();
    const childTools = result.tools;

    this.proxyTools = childTools.map(tool => this.augmentWithInstanceId(tool));
    this.proxyToolNames = new Set(childTools.map(t => t.name));  // NEW
    this.allTools = [...MANAGEMENT_TOOLS, ...this.proxyTools];
    this.initialized = true;
  }

  isProxyTool(name: string): boolean {
    return this.proxyToolNames.has(name);  // CHANGED from .some()
  }
}
```

Note: The Set is built from `childTools` (original names), not `this.proxyTools` (which have augmented schemas but same names). Either would work since `augmentWithInstanceId` preserves the name.
